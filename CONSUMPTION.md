# Consumption Contract & Release Versioning

> **Status:** canonical · **Issue:** [#22](https://github.com/Geoffe-Ga/aptitude-course/issues/22) · **Epic:** [adepthood#388](https://github.com/Geoffe-Ga/adepthood/issues/388)
>
> This document defines what the Adepthood app may rely on when it consumes
> this repository, how that surface is versioned, and how a content change
> travels from a merged PR to a deployed app. It pairs with
> [adepthood#391](https://github.com/Geoffe-Ga/adepthood/issues/391) (the
> app-side sync that pins a commit of this repo) and
> [adepthood#389](https://github.com/Geoffe-Ga/adepthood/issues/389) (the
> app-side manifest schema).

---

## 1. The published surface

The app vendors a **pinned commit** of this repository (the SHA lives in the
app's `CONTENT_VERSION`). Within that commit, the app may rely on exactly
three things:

| Surface | Contract |
|---------|----------|
| **`manifest.json`** | The machine-readable index, generated by `scripts/build_manifest.py` and validated against [`schema/manifest.schema.json`](./schema/manifest.schema.json). Carries `schema_version` (semver, see §2). The app keys all reads on this file. |
| **Markdown bodies** | The files under `markdown/**` that the manifest references via each entry's `path` — chapters, site resources, and, as of `schema_version` `1.1.0`, stage introductions (`manifest.json`'s optional `stage_intros[]`, CONTENT_FORMAT.md §3.3). Bodies are CommonMark per [CONTENT_FORMAT.md](./CONTENT_FORMAT.md) §2 — no raw HTML, renderable natively. |
| **Assets** | Images and other media referenced from those bodies or from `media[]` frontmatter, resolved per the asset convention (CONTENT_FORMAT.md §6, issue [#25](https://github.com/Geoffe-Ga/aptitude-course/issues/25)). Asset paths are repo-relative and stable. |

**Everything else is internal.** `google_docs/`, `prompts/`, `scripts/`,
`markdown/backup/`, `markdown/meta/`, per-stage `README.md`/TOC files, this
repo's CI configuration, and any file the manifest does not reference may be
renamed, moved, or deleted without notice. If the app reads it, that's a bug
on the app side. This is why the raw Google-Docs exports under `google_docs/`
were never a valid integration point for stage introductions — only the
Markdown they were converted to and indexed via `stage_intros[]` is
contracted.

Two corollaries:

- **Reads go through the manifest.** The app discovers chapters and site
  resources by iterating `manifest.json`, never by globbing `markdown/**`.
- **Identity is `id`** (and `slug` for site resources). Paths are stable in
  practice but the durable key for progress tracking, bookmarks, etc. is the
  frontmatter `id`, which is never reused or renumbered once shipped
  (CONTENT_FORMAT.md §3.2).

## 2. `schema_version` semver rules

`manifest.json` carries `schema_version` (currently `1.1.0`), the semver of
the **manifest contract** — not of the content. Bumps are coordinated with
adepthood#389, which owns the consuming schema.

**Precedent (`1.0.0` → `1.1.0`):** adding the optional `stage_intros[]` tier
(content-repo specs CR-A…CR-D, CONTENT_FORMAT.md §3.3, paired with
adepthood#717) is the first minor bump under this policy — it added one new
optional top-level array and touched nothing else, so a `1.0.0` consumer
built before the tier existed keeps validating and rendering the manifest
correctly; it simply never sees `stage_intros`.

| Bump | When | Examples |
|------|------|----------|
| **Patch** | Content-only changes. The set of fields, their types, and their semantics are untouched. | New chapters, edited bodies, reordered `order` values, new release tags. In practice patch releases don't even require a `schema_version` change — the version describes the contract, and most releases are content-only. |
| **Minor** | Additive, backwards-compatible contract changes. | A new **optional** frontmatter/manifest field; a new `content_type` enum value the app can safely ignore; a new optional top-level array. |
| **Major** | Breaking changes. | Removing or renaming a field; changing a type; tightening a pattern or rule that previously-valid consumers depended on; changing identity semantics (`id`, `(stage, chapter)`, slug rules). |

**App compatibility expectation:** the app accepts a manifest whose major
version equals the major it was built against, and must tolerate unknown
additive fields (minors). On a major mismatch the app must refuse the content
update rather than render undefined behavior — which is why majors are
coordinated, versioned, and shipped together with the app change
(see §4 and CONTENT_FORMAT.md §8).

**Not a version bump:** how the app *computes* lock/unlock state from the
published fields is app-side behavior, not part of this contract. As of
2026-07 the app derives a proportional drip schedule from `release_day`
(ordering), chapter count, and its own app-owned stage duration, rather than
gating solely on `release_day > days_elapsed` (see CONTENT_FORMAT.md §5.2).
`release_day`'s authored value, type, and meaning are unchanged, so this
shipped with **no** `schema_version` bump — it stayed at `1.1.0`.

## 3. Release tagging

The app pins a **tag** (resolved to its SHA), never a moving branch. Tags
follow a date-based convention:

```
content-vYYYY.MM.DD          e.g. content-v2026.06.09
content-vYYYY.MM.DD.N        N = 2, 3, … for additional cuts the same day
```

Rationale: content releases are a stream of editorial snapshots, not an API —
dates communicate recency better than a semver triple would. The semver that
matters for compatibility is `schema_version` *inside* the manifest (§2).

### Cutting a release

1. Ensure `main` is green (Content CI: frontmatter + manifest, lint, links).
2. Tag the commit and push the tag:

   ```bash
   git tag content-v$(date +%Y.%m.%d) && git push origin --tags
   ```

3. The tag is immutable once pushed — never move or delete a published tag;
   cut a new one instead.

Any green commit on `main` is eligible; in practice releases are cut when a
meaningful batch of content has landed.

## 4. The update handshake

How a content change reaches users:

```
content PR merged into main
        │
        ▼  Content CI green (frontmatter/manifest/lint/links — required checks)
tag content-vYYYY.MM.DD pushed
        │
        ▼  adepthood#391: bump CONTENT_VERSION to the new tag's SHA
app build vendors manifest.json + markdown/** + assets at that SHA
        │
        ▼  app deploy
users see the new content
```

Properties of this design:

- **Content ships independently of app deploys** — a content release is just
  a pin bump, with no app code change, for patch/minor content.
- **Every deployed corpus is auditable**: the pinned SHA identifies exactly
  what shipped, and CI guarantees the manifest at that SHA matched the files.
- **Rollback is a pin revert** to the previous tag's SHA.
- **Breaking (major) changes invert the order**: the app-side change
  (adepthood#389 schema + renderer) merges first behind the old pin, then the
  content major is tagged, then the pin and app ship together. Never bump the
  pin onto a major the deployed app doesn't understand.

---

## 5. Breaking-change policy

- The published surface (§1) is **never** broken without a **major**
  `schema_version` bump coordinated with adepthood#389 and a paired app
  change (§4).
- Field-level rules live in [CONTENT_FORMAT.md](./CONTENT_FORMAT.md) §8
  (change control); identity rules in §3.2 of the same document.
- Deprecations: prefer additive replacement (minor) + a deprecation note in
  the field's schema `description`, with removal deferred to the next major.
